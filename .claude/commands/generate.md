# 슬라이드 병렬 생성: $ARGUMENTS

`decks/<deck-name>/brief.md`의 슬라이드 맵을 읽어, **각 슬라이드를 장당 파일로 병렬 생성**한다.

## 진행

1. **brief.md 로드**: deck-name이 없으면 `decks/` 에서 찾는다(유일하면 그것, 여러 개면 사용자에게 확인). 맵 표 + **슬라이드 상세** + **용어집·톤**을 읽는다.

2. **병렬 렌더링**: 슬라이드마다 **서브에이전트(`executor`) 1명**을 띄워 동시에 작성한다. 서브에이전트는 **brief를 형식으로 옮기는 렌더러** — *내용·사실을 새로 지어내지 않는다*. (생각은 brief에서 이미 끝났다 → 장 사이 용어·톤·중복이 안 흔들림)
   - 슬라이드끼리 파일이 분리돼 있어 충돌 없음. 한 메시지에서 여러 에이전트를 동시에 띄운다.
   - 각 에이전트에 전달: **그 장의 '슬라이드 상세' 블록(실제 들어갈 말)** + 핵심 메시지 + **용어집·톤** + `patterns/PATTERNS.md`의 해당 레이아웃 스니펫.
   - 규칙: brief에 적힌 문구를 충실히 렌더한다. brief에 **없는 사실을 만들어내지 말 것** — 상세가 부족하면 최소한으로 형식만 잡고 `<!-- TODO: brief에 내용 필요 -->`로 표시해 보고한다(지어내기 ❌). 용어는 용어집 표기로 통일.
   - 각 에이전트는 `decks/<deck-name>/slides/NN-name.md` **하나만** 작성한다.

3. **2구역 형식 준수** (CLAUDE.md):
   - 위: 디자인 구역 — `<!-- _class: ... -->` 등 Marp 지시 + 필요한 HTML 래퍼.
   - 아래: 내용 구역 — 제목·불릿·코드(말). 한 슬라이드 한 메시지, 불릿 6개 이하.
   - 파일 안에 front matter(`---`)를 넣지 않는다(빌드가 슬라이드 구분자로 오인). 설정은 `_header.md`에만.

4. **빌드 검증**: 전부 생성되면 `node build.mjs <deck-name>` 실행, 에러 없이 `dist/<deck-name>.html` 생성 확인.
   - **`⚠ 넘침 위험` 경고가 뜬 슬라이드는 과밀**이다 → 해당 장 불릿/코드를 줄이거나 2단(`two`/`cols`)으로 분할해 경고가 사라질 때까지 다듬은 뒤 보고.

5. **검토 권유**: "초안 완성. `/content`·`/design`·`/slide`로 다듬거나 `/review <deck>`로 검토 패스를 돌리세요."
